## Root causes

1. `comfyui_generate.py` had no help branch, so `--help` and `-h` fell through as prompt text and reached workflow/ComfyUI execution.
2. Successful generation exposed only an unstructured `scene_<uuid>.png` line.
3. Native `generatePortrait()` ignored the exact subprocess output and rescanned the whole character directory for the newest `scene_*.png`; unrelated older files could therefore be promoted.
4. A zero subprocess exit code was treated as portrait success before durable character adoption was proven.
5. Portraits used a fixed `<id>_portrait.png` path, allowing Webview/browser image caching to retain old bytes after regeneration.
6. Antigravity instructions treated a printed path/visible image as sufficient and did not require character JSON adoption proof.
7. External Antigravity updates to `characters/<id>.json` had no bounded host watcher to refresh the left Character Profile list.

## Artifact authority

Successful generation now emits a prefixed JSON stdout record:

```text
TA_MEDIA_RESULT {"success":true,"outputPath":"...","createdAt":"...","characterId":"hero"}
```

The legacy plain absolute path is still printed for standalone compatibility, but LoreRelay portrait success requires the machine-readable record. The authoritative proof is:

1. `success` is true;
2. `outputPath` exists and is an allowed workspace image;
3. `createdAt` is valid and belongs to the current generation attempt;
4. returned `characterId` matches the requested character;
5. `characters/<id>.json` exists, belongs to that ID, and its `portrait` resolves to exactly `outputPath`.

An AI statement, command attempt, directory modification time, old chat image, or plain path line is not authority.

## Exact adoption flow

`comfyui_generate.py` accepts the optional paired flags:

```text
--character-id <id> --workspace <workspacePath>
```

When both are supplied:

1. ComfyUI produces one exact downloaded `scene_<uuid>.png`.
2. The adoption seam validates character ID, workspace, source existence, source type/size, non-symlink status, workspace containment, `characters/`, and matching character JSON identity.
3. The exact artifact is copied to a versioned character-owned path.
4. Character JSON is atomically replaced with that exact path.
5. Only after the JSON commit succeeds are older generated versions and the exact temporary native `scene_<uuid>.png` cleaned.
6. The successful `TA_MEDIA_RESULT` returns the adopted path and creation time.
7. Native `generatePortrait()` parses that exact result, independently verifies file/JSON/freshness, creates a safe Webview URI, posts `portraitGenerated`, refreshes `characterList`, and only then reports success.
8. A `characters/*.json` watcher refreshes the left list after an external Antigravity adoption.

No directory-wide “latest image” selection remains in the native portrait path.

## Cache strategy and cleanup

Adopted files use:

```text
characters/<characterId>_portrait_<16-hex-version>.<ext>
```

The version token is derived from the generated bytes plus the generation timestamp. A regeneration therefore produces a new filesystem path and a new Webview URI rather than reusing cached bytes under a fixed URI.

After successful JSON adoption, cleanup removes only older files matching the same character-owned versioned naming pattern. It does not broadly delete `scene_*.png`, other characters' files, fixed legacy/user portraits, symlinks, or files outside `characters/`. The exact native temporary `scene_<uuid>.png` is removed only when it is the source just adopted. Cleanup never runs before the authoritative JSON update.

## Failure behavior

- Generation failure: no adoption call; character JSON and previous portrait remain unchanged.
- Missing/outside/unsupported artifact: adoption fails before mutation.
- Missing, unreadable, or wrong-character JSON: adoption fails before mutation.
- Copy or atomic JSON failure: any newly copied destination is removed; previous JSON/portrait remains authoritative.
- Subprocess exit zero without a valid/fresh result or exact JSON binding: host reports verification failure and does not show portrait success.
- Adoption failure emits `success:false`, a technical error, and a nonzero exit; it never promotes an old image.

## Antigravity Skill behavior

The repo-owned Skill now requires:

- `--help`/`-h` are never prompts;
- generation and adoption use one command with character/workspace flags;
- success requires `TA_MEDIA_RESULT`, an existing adopted path, matching character ID, and exact character JSON binding;
- old `scene_*.png`, a previous chat image, or an attempted command cannot prove success;
- no newest-file guessing or stale-image reuse;
- no invented `file:///` Markdown paths;
- no stale portrait path written into `turn_result.json`;
- filesystem plus validated character JSON is authority, not AI narration.

Because this task explicitly forbids running the installer, the normal source test no longer assumes a developer machine's previously installed Skill must already hash-match the changed repo source. Exact installed-source hash enforcement remains available for explicit installation verification with `LORERELAY_REQUIRE_INSTALLED_SKILL_SYNC=1`.

## Human smoke steps

### A. CLI help

Run:

```text
python antigravity-skill/text-adventure-gm/scripts/comfyui_generate.py --help
python antigravity-skill/text-adventure-gm/scripts/comfyui_generate.py -h
```

Expected: real usage text, exit code 0, no workflow error, and no ComfyUI request.

### B. LoreRelay native portrait

1. Open an existing character in Character Profile.
2. Note the current portrait path/image.
3. Click Generate Portrait with a valid MEDIA-M1 profile.
4. Inspect `LoreRelay: Image Gen` output.

Expected:

- output contains one successful `TA_MEDIA_RESULT`;
- its `outputPath` is `characters/<id>_portrait_<version>.png` (or another allowed extension);
- `characters/<id>.json` points to that exact path;
- the left Character Profile refreshes to the new image;
- no unrelated old `scene_*.png` is selected.

### C. Regeneration/cache

Generate the same character again.

Expected: a different versioned path and URI are used, the new image appears without showing cached old bytes, the previous generated version is removed, and other files are preserved.

### D. Antigravity/manual portrait

Run the repo-owned Skill command shape with the intended character and workspace:

```text
python comfyui_generate.py "portrait prompt" "<workspace>\characters" illustrious --character-id <id> --workspace "<workspace>"
```

Expected: Antigravity claims success only after a successful `TA_MEDIA_RESULT` and matching character JSON. It does not embed an invented `file:///` image or reuse an older visible portrait. With LoreRelay open, the character JSON watcher refreshes the left character list.

### E. Adoption failure

Repeat with an invalid character ID or a workspace that lacks the matching character JSON.

Expected: `success:false`/nonzero exit, no success claim, no stale portrait promotion, and the previous valid character portrait remains unchanged.
